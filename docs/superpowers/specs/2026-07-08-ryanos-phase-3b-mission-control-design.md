# RyanOS — Phase 3B Design: Mission Control (tiled dashboard view)

**Date:** 2026-07-08
**Repo:** `~/Code/damon-ade` (RyanOS)
**Builds on:** Phase 2B-2 + 3A (shipped). The ADE app already has a **first-class browser/webview capability** — `BrowserPane` + `usePersistentWebview` (an `Electron.WebviewTag` registry keyed by paneId, hidden-container pattern so webviews persist across tab switches), a `browser-manager` in main, and browser tabs alongside terminal tabs. 3B does NOT build embedding — it curates existing dashboards into one screen using that capability.
**Goal:** A dedicated top-level **Mission Control** screen that tiles Ryan's live homelab/ops dashboards as a grid of persistent webviews — a "glance at everything" command center — with graceful handling of the very different reachability of each dashboard.

---

## 1. Scope

Phase 3 = 3A inter-agent handoff (shipped) + **3B dashboards-as-panels** (this spec). This spec is 3B only. It is mostly renderer work riding on the existing webview infra; the only main-process addition is a tiny config accessor.

## 2. The dashboard roster (config-driven, seeded with 5)

All five verified in the vault; reachability differs sharply and drives the UX:

| id | name | url | kind | reachability |
|---|---|---|---|---|
| `ops-deck` | Ops Deck | `http://192.168.86.43:8787` | lan | Core box, Docker 24/7 — up when on the home LAN |
| `rubypulse` | RubyPulse | `http://192.168.86.28:7420` | lan | laser PC — up when on LAN (verified HTTP 200) |
| `mypka` | myPKA Cockpit | `http://localhost:4317` | localhost | local server on this Mac — only when running (dies on reboot) |
| `catchpad` | CatchPad | `https://catchpad-dash.pages.dev` | web | Cloudflare Pages, **gated by Cloudflare Access** (login renders in the webview) |

The roster lives in a config (not hardcoded), so Ryan adds/removes/reorders without a rebuild. Each entry: `{ id, name, url, kind }`. `kind` is a hint for the status/empty-state copy (e.g. "LAN — are you home?" vs "local server not running").

## 3. Mission Control view

**Placement.** A new top-level surface, peer to the agents/workspace view — reached from a nav entry (in `WorkspaceSidebar` or a top-level switcher) and its own TanStack route under `routes/_authenticated/`. Switching to it shows the grid; switching away keeps the webviews warm (persistent registry), so returning is instant.

**Layout.** A responsive auto-grid of **tiles** (2×2 / 2×3 by count; CSS grid, `overflow` contained). v1 is auto-layout only. Each tile:
- **Title bar:** dashboard name + a **status dot** (loading / live / unreachable) + **refresh** (reload the webview) + **pop-out** (open the URL in a full browser tab / external).
- **Body:** a persistent webview for the dashboard URL.

**Per-tile status + reachability (the load-bearing UX).** Drive the status dot from the webview's own load events — `did-start-loading` → loading, `did-finish-load` → live, `did-fail-load` (or timeout) → unreachable. An unreachable tile renders a clean empty state (the name, the URL, a `kind`-aware hint, and a **Retry** button that reloads), never a broken-page mess. CatchPad's Cloudflare Access login simply renders inside its webview; the persistent session keeps Ryan logged in across visits. Code HQ loads from `file://`.

**Config access.** The roster is stored at `~/.ade/mission-control.json`, seeded from a default constant on first read. The renderer reads it via a small tRPC procedure (`missionControl.getDashboards`) — matching the app's tRPC-for-IPC convention. Ryan hand-edits the JSON for v1; in-app add/edit CRUD is v2.

## 4. Reuse of existing infra (what we DON'T build)

- **Webview embedding:** reuse `usePersistentWebview` / the `webviewRegistry` / hidden-container pattern (`.../TabView/BrowserPane/hooks/usePersistentWebview`). Mission Control creates one persistent webview per dashboard id (stable key = `mc:<dashboard.id>`), so they survive navigation like browser tabs do.
- **Routing/nav:** TanStack Router (add one route) + one nav entry.
- No `browser-manager`/main-process embedding changes; no new Electron windows.

## 5. What gets built

1. **Config:** a `DEFAULT_DASHBOARDS` constant (the 5 above) + a tRPC `missionControl` router that reads `~/.ade/mission-control.json` (seeding it from the default if absent) and returns the list. (Small main-process addition.)
2. **Route + nav entry:** a top-level Mission Control route under `routes/_authenticated/` + a sidebar/switcher entry to reach it.
3. **`MissionControlView`** (renderer screen): fetches the roster via tRPC, renders the responsive tile grid.
4. **`DashboardTile`** component: title bar (name + status dot + refresh + pop-out) + a persistent webview (reusing `usePersistentWebview`, keyed by `mc:<id>`) + the unreachable/loading/live states driven by webview load events.
5. Wire load-event → status + the Retry-reloads behavior.

Mostly renderer; one small tRPC router. No changes to agents, seed, or `agent-scaffold`/`seed-brains` — fully disjoint from the concurrent SM Team (`assets/seed-brains/`) and 3A work.

## 6. Verification

- Launch the app, open **Mission Control**: all 5 tiles render in a grid.
- **RubyPulse** + **Code HQ** tiles go **live** (LAN 200 + local file) — visible content.
- **myPKA** (if `localhost:4317` down) and **Ops Deck** (if off-LAN) show the **unreachable + Retry** state cleanly, not a crash; starting the myPKA server + Retry flips it to live.
- **CatchPad** renders the Cloudflare Access login (or the dashboard if already authed).
- Switch away to an agent and back → tiles are still warm (persistent webviews), no full reload.
- Edit `~/.ade/mission-control.json` (reorder / add a tile) → after reload, the grid reflects it.
- Pop-out on a tile opens the dashboard full/externally.

Success = one screen where Ryan glances at every dashboard, dead tiles fail gracefully with Retry, and the roster is a file he can edit.

## 7. Non-goals (later)

- Drag-to-rearrange / resizable tiles (v1 is auto-grid).
- In-app add/edit/remove dashboards (v1 edits the JSON; CRUD is v2).
- Auth automation for Access-gated tiles (the webview login is fine).
- Embedding non-web things, custom per-dashboard adapters, or richer "Code HQ awareness" beyond the `dashboard.html` tile.
- Auto-starting the myPKA local server (out of scope; unreachable-state + Retry covers it).

## 8. Risks

- **Multiple live webviews cost memory/CPU.** Five persistent `<webview>`s is modest, but note it; if it bites, add a "pause offscreen tiles" option later (v1 keeps them warm for instant return).
- **Reachability is genuinely flaky** (LAN when away, localhost when down, Access-gated). This is why per-tile graceful states + Retry are core, not polish — the view must be useful with only 2–3 tiles live.
- **`file://` in a webview** may need `webPreferences`/allowFileAccess on the webview tag; confirm Code HQ's `dashboard.html` loads (it's self-contained, so likely fine).
- **Cloudflare Access in a webview** — the login must complete inside the embedded webview; if it hard-blocks embedding (X-Frame-Options/CSP), fall back to pop-out for that tile. Verify during build; not a blocker (pop-out always works).
- **Config path assumes `~/.ade`** (the app data dir via `getSupersetHomeDir()`); use that helper, not a hardcoded home path, so an `ADE_HOME_DIR` override still works.
